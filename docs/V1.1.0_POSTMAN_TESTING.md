# Version v1.1.0 Postman & Production Testing Guide

Set a Postman environment variable named `baseUrl` to the deployed API host, for example `http://localhost:5000`. The API routes continue to use the `/api` prefix; health endpoints are intentionally outside it for load-balancer use.

## Health checks

1. Send `GET {{baseUrl}}/health/live`.
2. Expect `200 OK`, `success: true`, `data.status: "live"`, and an ISO timestamp. This endpoint does not require MongoDB, SMTP, or Cloudinary.
3. Send `GET {{baseUrl}}/health/ready`.
4. Expect `200 OK` only when MongoDB is connected, Cloudinary credentials are present, and the SMTP transport verification succeeds. The response includes the state of each dependency and never includes credentials.
5. To verify the negative path, run an instance with unreachable SMTP/MongoDB or missing Cloudinary values. `/health/ready` must return `503` and `success: false`; `/health/live` must remain `200`.

## Swagger UI

Open `{{baseUrl}}/api/docs` in a browser. The documentation groups existing Authentication, Attendance, Leave, Employee, Department, Dashboard, Reports, Notifications, Health, document upload, and profile-picture endpoints. Use **Authorize** to provide an access JWT as `Bearer <token>` for protected endpoints. No token values are recorded in application logs.

## Startup validation

Production requires these values: `PORT`, `MONGO_URI`, `JWT_SECRET`, `JWT_REFRESH_SECRET`, `NODE_ENV`, `FRONTEND_URL` (or the supported legacy `CLIENT_URL`), all three Cloudinary variables, and either `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS` or legacy `EMAIL_USER`/`EMAIL_PASS`.

Run the server with one required value omitted. It must terminate before listening and emit a single meaningful `Server startup failed` log entry naming the missing configuration. Test invalid `PORT`, invalid `FRONTEND_URL`, and an unsupported `NODE_ENV` similarly.

## Graceful shutdown

1. Start the API with a reachable MongoDB and valid environment.
2. Make a long-running request if your deployment has one, then send `SIGTERM` (Linux/container) or `SIGINT` (Ctrl+C locally).
3. Confirm logs show shutdown started, cron task scheduling stopped, MongoDB closed, and shutdown completed.
4. Confirm the process exits with code `0` and the platform/load balancer removes the instance after readiness fails or the listener closes.

## Production configuration checklist

- Set `NODE_ENV=production`, a HTTPS `FRONTEND_URL`, and strong, unique JWT secrets.
- Use `SMTP_*` credentials from your secret manager; `SMTP_SECURE=true` for implicit TLS (typically port 465).
- Set Cloudinary credentials only through the deployment secret store.
- Keep frontend origin exact: CORS allows only the configured frontend and supports credentialed cookies.
- Deploy behind one trusted reverse proxy or adjust `trust proxy` deliberately for your topology. Cookies are `httpOnly`, `secure` in production, and `sameSite=lax`.
- Verify body requests above 1 MB receive `413`, auth abuse receives `429`, and security headers are present.
